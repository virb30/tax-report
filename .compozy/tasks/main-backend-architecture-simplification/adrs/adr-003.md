# ADR-003: Migrate Portfolio IPC to IpcResult

## Status

Accepted

## Date

2026-04-26

## Context

The current IPC layer mixes thrown renderer errors with result-shaped responses. Most portfolio
contracts use `errorMode: 'throw'`, while some broker mutations return `{ success: false, error }`.
The PRD requires consistent backend error feedback for validation, missing data, conflicts,
business-rule failures, and unexpected errors.

Portfolio workflows currently touch the affected renderer surfaces and give the MVP a narrow place
to standardize failure semantics before manual correction behavior is implemented in Phase 2.

## Decision

Convert portfolio IPC APIs to a shared `IpcResult<T>` response shape during the MVP. The shared
error model uses domain/application `AppError` values for known failures and maps unknown failures
to a generic unexpected error. Zod payload failures become validation results.

Existing portfolio APIs return `IpcResult<T>` consistently. Manual correction APIs are deferred to
Phase 2 and should use the same result model when added. Import, report, app, and broker APIs can
migrate in later phases unless they are directly touched by this MVP.

## Alternatives Considered

### Alternative 1: Defer IpcResult until correction APIs exist

- **Description**: Keep current portfolio IPC behavior in MVP and introduce `IpcResult` only when
  correction endpoints are added in Phase 2.
- **Pros**: Smaller MVP implementation.
- **Cons**: Leaves mixed portfolio semantics and forces Phase 2 to solve infrastructure before
  correction behavior.
- **Why rejected**: MVP should reduce backend failure risk before adding correction behavior.

### Alternative 2: Convert every renderer-facing API in MVP

- **Description**: Migrate portfolio, imports, reports, brokers, and app channels together.
- **Pros**: Full consistency across the application.
- **Cons**: Large blast radius unrelated to year-end correction value.
- **Why rejected**: The PRD favors incremental simplification over a broad rewrite.

### Alternative 3: Keep throw-based portfolio IPC

- **Description**: Keep portfolio APIs on the existing throw behavior.
- **Pros**: Minimal infrastructure change.
- **Cons**: Does not advance backend error consistency.
- **Why rejected**: The MVP explicitly focuses on portfolio backend error consistency.

## Consequences

### Positive

- Portfolio workflows receive predictable success and failure envelopes.
- The renderer can branch on typed failures instead of parsing thrown errors.
- The shared error/result primitives establish the later migration path for imports and reports.

### Negative

- Portfolio renderer hooks and tests need updates to unwrap `IpcResult`.
- Existing contract types must change for portfolio APIs.

### Risks

- Migrating all portfolio channels at once can miss a renderer call site. Mitigation: update the
  `ElectronApi` type, contract tests, and affected renderer tests together.
- Unknown errors may hide useful developer context from the UI. Mitigation: keep generic renderer
  messages but allow main-process logs or test assertions to retain details where appropriate.

## Implementation Notes

- Add shared `AppError`, `AppErrorKind`, and `IpcResult<T>` types.
- Extend IPC binding to produce result envelopes for result-mode contracts.
- Convert portfolio contracts and handlers to result mode.
- Update `usePositionsPage`, initial balance flows, consolidated position modal, and related tests
  to unwrap portfolio results.
