# ADR-001: Prioritize Portfolio Correction Auditability

## Status

Superseded by ADR-004

## Date

2026-04-26

## Context

The backend architecture simplification effort has three product outcomes: make backend feature
work easier to navigate, reduce user-facing defect risk through consistent backend errors, and
clarify manual position corrections for annual tax workflows. The design input recommends an
incremental migration rather than a broad rewrite. During PRD clarification, the selected rollout
shape was IPC and error standardization first, and the selected manual adjustment behavior was a
year-end override for the selected year.

The PRD needs one MVP focus so the scope does not become an architecture clean-up without a clear
user outcome.

## Decision

Prioritize portfolio correction auditability for the MVP. The MVP will define manual position
adjustments as year-end overrides for the selected year and use backend error consistency and
incremental architecture simplification to support that workflow.

## Alternatives Considered

### Alternative 1: Error-first simplification with explicit product semantics

- **Description**: Standardize backend operation outcomes and failure behavior first, while
  defining manual adjustments and migrating modules incrementally.
- **Pros**: Broadly improves reliability and gives the renderer a consistent failure contract.
- **Cons**: User-visible portfolio correction value may be delayed.
- **Why rejected**: It is a strong supporting direction, but the PRD needs the MVP to lead with a
  concrete tax workflow outcome.

### Alternative 2: Capability-first vertical modules

- **Description**: Establish portfolio, imports, and tax module ownership early, then standardize
  errors as each capability moves.
- **Pros**: Improves backend navigation quickly and makes business ownership visible.
- **Cons**: Higher migration blast radius and slower direct user value.
- **Why rejected**: The chosen rollout favors lower-risk standardization first, not broad module
  movement.

### Alternative 3: Error-only rollout

- **Description**: Focus the MVP only on IPC and backend error consistency, deferring manual
  adjustment semantics and module movement.
- **Pros**: Smallest initial scope and clear user-facing failure behavior.
- **Cons**: Leaves the most important product ambiguity unresolved.
- **Why rejected**: The year-end correction behavior is required to make portfolio corrections
  understandable and auditable for users.

## Consequences

### Positive

- The MVP has a concrete user outcome: reliable, auditable year-end portfolio corrections.
- Backend error consistency supports a visible workflow instead of existing as isolated polish.
- Incremental architecture simplification can be justified by product-facing correction and tax
  reporting needs.

### Negative

- General backend navigation improvements will arrive gradually.
- Some broader import and tax module simplification may remain outside the MVP.

### Risks

- Users may later expect date-forward corrections for transaction-level reconciliation. Mitigation:
  document date-forward behavior as out of scope for MVP and preserve room for a later adjustment
  type.
- Error standardization could expand beyond the correction flow. Mitigation: prioritize operations
  that affect portfolio correction, import preview, and annual reporting confidence.

## Implementation Notes

Implementation details belong in the follow-up TechSpec. The PRD should remain focused on user
capabilities, expected behavior, rollout scope, and success metrics.
