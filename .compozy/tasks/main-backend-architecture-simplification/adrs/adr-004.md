# ADR-004: Defer Portfolio Corrections To Phase 2

## Status

Accepted

## Date

2026-04-26

## Context

The PRD originally led with manual year-end portfolio corrections. During technical design, the
scope grew to include new correction persistence, projection ordering, audit history queries,
renderer flows, and portfolio-wide IPC result migration. That made the MVP carry both user-facing
correction behavior and backend infrastructure cleanup.

The team decided to simplify the MVP. Backend error consistency and ownership cleanup should land
first, while correction behavior moves to a follow-up phase.

## Decision

Move portfolio correction logic out of MVP and into Phase 2. MVP will not implement
`position_adjustments`, correction projection rules, correction audit UI, or annual report override
behavior.

The MVP will focus on portfolio IPC result standardization and incremental backend simplification.
Phase 2 will implement manual year-end corrections as explicit position adjustments, with
transaction/import recalculation applied first and adjustments applied afterward as the final
year-end override layer.

## Alternatives Considered

### Alternative 1: Keep correction auditability as MVP focus

- **Description**: Implement position adjustments, correction projection, audit reads, and
  portfolio IPC result migration together.
- **Pros**: Delivers direct tax-preparation correction value immediately.
- **Cons**: Larger MVP with more data model, projection, IPC, and renderer changes.
- **Why rejected**: The MVP should reduce backend risk before adding a new correction model.

### Alternative 2: Implement correction storage only

- **Description**: Add `position_adjustments` persistence without renderer correction flows.
- **Pros**: Starts the data model early.
- **Cons**: Creates unused storage and still requires projection decisions.
- **Why rejected**: Storage without the full correction workflow adds maintenance cost without user
  value.

### Alternative 3: Keep current backend behavior unchanged

- **Description**: Defer both corrections and error consistency.
- **Pros**: No near-term migration cost.
- **Cons**: Leaves mixed IPC failures and backend ownership friction in place.
- **Why rejected**: The task exists to simplify the backend before future feature work.

## Consequences

### Positive

- MVP scope becomes smaller and easier to verify.
- Portfolio correction work can build on a cleaner IPC error boundary in Phase 2.
- Existing portfolio and annual report behavior remains stable in Phase 1.

### Negative

- Users do not receive manual year-end correction behavior in MVP.
- Phase 2 still needs data model, projection, and audit UI design work.

### Risks

- Phase 2 may revisit decisions already discussed during this TechSpec. Mitigation: preserve the
  deferred correction notes in ADR-002 and this ADR.
- MVP could become infrastructure-only. Mitigation: keep the MVP tied to observable portfolio
  failure behavior and contract tests.

## Implementation Notes

- Do not create `position_adjustments` in MVP.
- Do not add correction IPC contracts or renderer correction flows in MVP.
- Keep `positions` and annual report behavior unchanged in MVP.
- Convert portfolio IPC APIs to `IpcResult<T>` as the main MVP implementation.
