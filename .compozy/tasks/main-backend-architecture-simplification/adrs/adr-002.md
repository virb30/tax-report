# ADR-002: Store Corrections as Position Adjustments

## Status

Superseded by ADR-004

## Date

2026-04-26

## Context

The MVP needs manual year-end corrections that are auditable and become the source of truth for
annual declaration output. The current backend stores year-scoped positions and reuses
`initial_balance` transactions for manual state, but a final declaration correction is not a
market transaction and should not require users to model synthetic ledger activity.

The PRD also requires the final-position-per-year screen to expose correction state without a
broad backend folder migration.

## Decision

Add a dedicated `position_adjustments` persistence model for correction intent. The table stores a
domain-controlled `type` string, with MVP records using `manual`. A saved manual adjustment
represents the final year-end quantity and average price for a ticker, broker, and year.

When the application saves a manual adjustment, it immediately reprojects and persists the
affected `positions` cache so `listPositions` and annual report generation read corrected year-end
state from the existing position query path. Projection order is explicit: recalculate all
transactions and imported position effects first, then apply position adjustments as the final
year-end override layer.

For MVP audit exposure, `listPositions` returns a `hasAdjustment` flag per position, and the
portfolio module exposes a read flow to list existing adjustments for a selected asset and broker.

## Alternatives Considered

### Alternative 1: Reuse transactions for manual corrections

- **Description**: Add a transaction type for manual correction and treat it specially in
  recalculation.
- **Pros**: Reuses the existing transaction repository and event flow.
- **Cons**: Mixes final declaration overrides with date-forward market events.
- **Why rejected**: It obscures user correction intent and weakens the audit model.

### Alternative 2: Store correction fields directly on positions

- **Description**: Add corrected quantity, average price, reason, and timestamp columns to
  `positions`.
- **Pros**: Smallest query and persistence change.
- **Cons**: Makes `positions` both cache and correction source of truth.
- **Why rejected**: It loses correction history and makes future adjustment types harder to
  represent.

### Alternative 3: Apply adjustments only at query time

- **Description**: Persist adjustments separately and overlay them in list/report queries.
- **Pros**: Preserves imported/calculated positions unchanged.
- **Cons**: Duplicates correction overlay rules across queries and reports.
- **Why rejected**: Immediate projection keeps the existing reporting path stable and makes the
  corrected cache explicit.

## Consequences

### Positive

- Correction intent is auditable and separate from market transactions.
- Annual reports use the existing `positions` read path while honoring year-end corrections.
- The MVP can ship without a broad `src/main` module migration.


### Negative

- A new table and repository boundary are required.
- The positions screen needs an additional detail query when the user inspects an adjusted
  ticker/broker pair.

### Risks

- Free-text adjustment types could drift without domain control. Mitigation: define allowed type
  constants in the portfolio domain and validate before persistence.
- Immediate projection can conflict with later recalculation if recalculation applies adjustments
  before transactions. Mitigation: route correction save and recalculation through the same
  transaction-first, adjustment-last projector behavior.

## Implementation Notes

- Use a migration named after the next database sequence to create `position_adjustments`.
- Keep the table separate from `transactions`.
- Store adjustment rows with `id`, `ticker`, `year`, `broker_id`, `type`, `target_quantity`,
  `target_average_price`, optional `reason`, and `created_at`.
- Add repository methods to save the latest adjustment, detect whether a position has any
  adjustment for a given year, and list adjustments by ticker, year, and broker.
