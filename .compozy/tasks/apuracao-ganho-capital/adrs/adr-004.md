# ADR-004: Current-Year Assessment With Corporate Action Support

## Status

Accepted

## Date

2026-05-04

## Context

The PRD requires supported B3 spot-market stocks, FIIs, and ETFs to be assessed month by
month, with same-category loss compensation and visible blockers for unsupported or
ambiguous cases.

Existing transactions include buys, sells, initial balances, bonus events, splits, reverse
splits, transfers, and fraction auctions. These events can affect quantity and average
cost. Treating every non-buy/sell row as unsupported would reduce useful coverage, while
ignoring them would risk incorrect cost basis.

The repository already has migrations for `accumulated_losses`, but no active user
workflow or repository coverage was found for prior-year loss management.

## Decision

V1 will calculate loss compensation only within the selected tax year and will defer
prior-year opening loss balances to a later phase.

V1 will support cost-basis effects for existing non-buy/sell transaction types that can
affect supported spot-market positions: initial balance, bonus, split, reverse split,
transfers, and fraction auction. Unsupported asset classes, day trade detection,
derivatives, options, futures, term market, securities lending, foreign assets, BDRs, and
ambiguous data remain blockers.

If a corporate action or transfer lacks enough data to produce a reliable average-cost
effect, the affected month must be marked `pending` or `mixed` with a blocker instead of
silently calculating a ready result.

## Alternatives Considered

### Alternative 1: Current-Year Buy/Sell Only

- **Description**: Calculate only buy and sell rows and block all other transaction types.
- **Pros**: Smaller implementation and simpler test matrix.
- **Cons**: Many real portfolios would become pending even when the imported data can
  support cost-basis continuity.
- **Why rejected**: The user selected corporate-action support for V1.

### Alternative 2: Ignore Non-Buy/Sell Rows

- **Description**: Calculate gains from buy and sell rows while omitting other transaction
  types from blockers.
- **Pros**: Fastest implementation.
- **Cons**: Can produce materially wrong average cost and false ready statuses.
- **Why rejected**: This violates the PRD's requirement to prevent false confidence.

### Alternative 3: Prior-Year Loss Entry in V1

- **Description**: Add UI/API fields or use `accumulated_losses` as opening balances.
- **Pros**: More complete compensation for users with historical losses.
- **Cons**: Expands product scope into persistent tax memory without an existing workflow.
- **Why rejected**: The MVP should prove annual assessment first; prior-year loss entry is
  a Phase 2 candidate.

## Consequences

### Positive

- Produces more useful ready months for portfolios with supported corporate actions.
- Avoids treating unused `accumulated_losses` data as reliable opening tax memory.
- Keeps false-confidence protection by blocking ambiguous cost-basis cases.

### Negative

- Increases the domain service and fixture test matrix.
- Current-year results may understate available compensation for users with prior-year
  losses until Phase 2 adds opening balance support.

### Risks

- **Corporate action mistakes**: Mitigate with unit tests for quantity and average-cost
  effects before and after sales.
- **Prior-year loss confusion**: Mitigate by exposing that V1 compensation starts at zero
  for the selected year.
- **Unsupported day trade leakage**: Mitigate by detecting same-day buy/sell patterns and
  surfacing them as blockers.

## Implementation Notes

- Start each supported category loss balance at zero for the selected year.
- Carry losses forward month by month only within the selected year.
- Keep supported categories separate for stock, FII, and ETF compensation.
- Add tests for threshold equality at R$ 20,000, FIIs/ETFs without stock exemption,
  corporate actions before sales, and ambiguous events that must block readiness.
