# ADR-003: Tax Reporting Read Model for Capital Gains Assessment

## Status

Accepted

## Date

2026-05-04

## Context

The capital gains assessment must consume existing transactions, transaction fees, asset
classifications, daily broker taxes, and tax-year data without changing import,
portfolio, asset catalog, positions, or Bens e Direitos behavior.

The existing `tax-reporting` context already generates annual reporting output by reading
portfolio data and returning renderer-ready DTOs. The new assessment needs a broader
joined view than a single aggregate repository can provide, including monthly grouping,
asset-specific classification, blockers, and calculation traces.

## Decision

Implement the V1 capital gains assessment as an additive use case inside
`src/main/tax-reporting`.

The use case will depend on a dedicated read/query port in the tax-reporting application
layer, backed by a Knex adapter in the tax-reporting infra layer. The query will return
assessment source DTOs for the selected tax year, including transactions, allocated fees,
asset classifications, and daily broker tax data needed for monthly assessment.

The backend use case will return render-ready monthly rows, annual totals, blockers, and
sale-level trace DTOs through a new typed IPC contract. The renderer will display the
assessment in a dedicated Renda Variavel page/tab.

## Alternatives Considered

### Alternative 1: New Capital Gains Bounded Context

- **Description**: Create a separate `capital-gains` context with its own application,
  domain, infra, and transport layers.
- **Pros**: Strong conceptual separation and future room for DARF workflows.
- **Cons**: More structure than V1 needs; duplicates existing report transport and
  container patterns; increases implementation surface before the workflow proves itself.
- **Why rejected**: V1 is an additive assessment/reporting workflow, so the existing
  `tax-reporting` context is the smaller and clearer boundary.

### Alternative 2: Extend Portfolio Repositories

- **Description**: Add reporting-specific methods to existing portfolio and ingestion
  repositories, then compose them in the use case.
- **Pros**: Reuses existing repository abstractions.
- **Cons**: Pushes report-shaped joins into aggregate repositories; increases coupling
  between portfolio persistence and tax reporting needs.
- **Why rejected**: The assessment is a read model, not aggregate persistence.

### Alternative 3: Renderer-Side Assessment Assembly

- **Description**: Return raw facts through IPC and calculate monthly rows in React.
- **Pros**: Keeps backend output thin.
- **Cons**: Moves tax rules into the renderer, weakens testability, and risks inconsistent
  annual totals and traces.
- **Why rejected**: Tax assessment rules belong in backend application/domain code with
  focused Jest coverage.

## Consequences

### Positive

- Keeps the feature additive and localized to tax reporting.
- Preserves existing portfolio and ingestion responsibilities.
- Makes the monthly assessment contract explicit and testable.
- Gives the renderer a stable, display-oriented API.

### Negative

- Adds a new query adapter instead of using only existing repositories.
- The read model must be kept aligned with schema changes in transactions, fees, assets,
  and daily broker taxes.

### Risks

- **Read model drift**: Mitigate with query adapter tests and contract tests around the IPC
  shape.
- **Over-large DTOs**: Mitigate by returning only sale-level trace fields needed by the MVP
  table and expansion view.
- **Context creep**: Mitigate by keeping DARF generation, paid-tax memory, and official
  filing out of this use case.
