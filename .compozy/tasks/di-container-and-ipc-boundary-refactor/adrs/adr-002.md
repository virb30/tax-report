# ADR-002: Define a Dedicated IPC Public API Module and Remove Renderer Access to Shared Domain Types

## Status

Accepted

## Date

2026-05-05

## Context

The current `src/preload` tree mixes four concerns: public IPC contracts, renderer API typing,
main-process IPC binding/registry infrastructure, and the Electron preload bridge runtime. At the
same time, the renderer imports both deep IPC contract files and domain/shared enums from
`src/shared/types/domain.ts`.

The desired boundary is closer to an HTTP-style API surface: request/response DTOs are public,
while domain/application models remain internal to the backend.

## Decision

Create a dedicated IPC public API module, separate from `src/shared`, and make the renderer depend
only on that public entrypoint. Preserve the existing IPC channel names and method semantics, but
stop exposing deep file paths and shared domain enums as renderer dependencies.

`src/preload` remains responsible only for Electron preload runtime concerns such as
`contextBridge` and `ipcRenderer` exposure. Public transport DTOs, renderer-facing IPC typings,
and transport schemas move under the dedicated IPC module.

## Alternatives Considered

### Alternative 1: Move contracts into `src/shared`

- **Description**: Treat IPC contracts as shared code under the existing shared root.
- **Pros**: Aligns with "code used by more than one process" at a filesystem level.
- **Cons**: Keeps API DTOs conceptually mixed with domain/shared utility concerns.
- **Why rejected**: The goal is to distinguish public IPC API from domain-shared code, not to
  strengthen the current shared bucket.

### Alternative 2: Keep `src/preload` as the public API package

- **Description**: Preserve the current root and only improve exports.
- **Pros**: Lower migration effort.
- **Cons**: Continues mixing Electron runtime and public API ownership.
- **Why rejected**: The current naming and package shape obscure the process boundary.

## Consequences

### Positive

- Renderer imports become stable and intentional.
- IPC DTOs are treated as the public API surface between processes.
- Domain/internal types stop leaking into the renderer.

### Negative

- The migration touches many imports in renderer tests and pages.
- Some enums and DTO names may need local duplication across backend and public transport layers.

### Risks

- A direct-cut migration can break many files at once. Mitigation: preserve channel names and keep
  a single new public entrypoint so all import churn targets one package.

## Implementation Notes

- Publish one renderer-facing entrypoint for transport DTOs and API typing.
- Keep channel names unchanged.
- Remove renderer imports from `src/shared/types/domain.ts`.
