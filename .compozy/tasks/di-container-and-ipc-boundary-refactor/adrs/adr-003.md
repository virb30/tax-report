# ADR-003: Keep Application Use-Case Inputs and Outputs Inside `src/main`

## Status

Accepted

## Date

2026-05-05

## Context

Several application use-cases import request or response types directly from IPC contract files.
Examples include `save-initial-balance-document.use-case.ts` and `preview-import.use-case.ts`.
This makes application behavior depend on the transport-layer API shape and prevents the transport
boundary from acting as a true adapter.

The selected architecture direction treats IPC handlers like controllers in a traditional web
application: they own request/response mapping at the boundary.

## Decision

Application use-cases and application services will define and export their own input/output types
inside `src/main`. IPC handlers and registrars will map between public IPC DTOs and
application-layer models.

This applies even when the public DTO and the application DTO are structurally similar. Similarity
alone is not a reason to share ownership across layers.

## Alternatives Considered

### Alternative 1: Keep using IPC DTOs directly in use-cases

- **Description**: Leave use-case signatures coupled to public contracts.
- **Pros**: Less mapping code.
- **Cons**: Transport changes leak into application code and make the public API the de facto
  application model.
- **Why rejected**: It violates the intended architecture boundary.

### Alternative 2: Share a single DTO package between IPC and application

- **Description**: Create one model package to serve both public transport and application code.
- **Pros**: Fewer duplicate type declarations.
- **Cons**: Collapses two layers into one ownership boundary.
- **Why rejected**: The project explicitly wants IPC DTOs to be treated as boundary contracts, not
  as application types.

## Consequences

### Positive

- Application use-cases become independent from Electron IPC.
- Contract evolution becomes easier because mapping lives at the boundary.
- Tests can target application behavior without importing transport DTOs.

### Negative

- Additional mapping code is required in handlers.
- Some structurally similar types will exist in both transport and application layers.

### Risks

- Duplicate shape definitions can drift. Mitigation: keep handlers thin and add transport-to-
  application mapping tests around the affected IPC handlers.

## Implementation Notes

- Export `Input` and `Output` types from each use-case file, following repository conventions.
- Keep boundary mapping inside transport handlers/factories, not inside domain or application
  services.
