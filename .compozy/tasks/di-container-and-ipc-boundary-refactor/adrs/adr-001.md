# ADR-001: Keep Awilix and Split the Main Composition Root by Context

## Status

Accepted

## Date

2026-05-05

## Context

The current main-process dependency injection setup is centered in
`src/main/app/infra/container/index.ts`. That file currently wires repositories, parsers,
services, use cases, IPC registrars, and registry aggregation for multiple bounded contexts. It
also exports a global `container` singleton and resolves local dependencies through
`container.resolve(...)` inside registration closures.

This shape creates a bottleneck for changes across `portfolio`, `ingestion`, `tax-reporting`, and
`app` concerns. The refactoring goal is to reduce container sprawl without paying the migration
cost of replacing the DI framework.

## Decision

Keep Awilix, but replace the single global composition root with context-level registration
modules. The new root bootstrap will create a container per app start, register shared
infrastructure, invoke module registrars for each bounded context, and expose only the assembled
bootstrap outputs needed by `main.ts`.

The design explicitly removes the exported global `container` and avoids `container.resolve(...)`
as an internal service-locator pattern inside registration functions.

## Alternatives Considered

### Alternative 1: Replace Awilix with full manual dependency injection

- **Description**: Remove the DI framework and build all use cases, services, and registrars with
  explicit factories.
- **Pros**: Maximum explicitness and no container indirection.
- **Cons**: Larger migration, broader blast radius, and more boilerplate in this codebase.
- **Why rejected**: The main problem is composition ownership, not the existence of Awilix.

### Alternative 2: Keep the current global container and only reorder registrations

- **Description**: Leave the file in place and apply small cleanup within the same module.
- **Pros**: Smallest implementation effort.
- **Cons**: Does not solve divergent change, hidden dependency resolution, or cross-context
  ownership ambiguity.
- **Why rejected**: The structural bottleneck remains.

## Consequences

### Positive

- Each bounded context owns its composition rules.
- The app bootstrap becomes smaller and easier to reason about.
- Tests can target module registrars instead of one global registration file.

### Negative

- Registration code moves across several files.
- Some Awilix convenience patterns need to be replaced with more explicit registration helpers.

### Risks

- Shared dependencies such as database connection and queue adapters can become duplicated across
  modules. Mitigation: define a small shared infrastructure registration layer used by all module
  registrars.

## Implementation Notes

- Introduce context-level registration modules under each context's infra layer.
- Add a root `createMainBootstrap` or equivalent bootstrap factory in `src/main/app`.
- Keep singleton lifetimes where current behavior depends on them.
