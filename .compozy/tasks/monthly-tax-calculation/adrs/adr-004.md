# ADR-004: Persist Monthly Close Artifacts and Recalculate Forward Automatically

## Status

Accepted

## Date

2026-05-07

## Context

The PRD requires monthly tax close to be a trust-first workflow with explicit month states, month-level auditability,
and full-history recalculation. Monthly results can change when upstream facts change, including imported transactions,
daily broker taxes, IRRF values, and asset classifications. Those changes also affect later months because losses,
IRRF credits, and below-minimum carry-forward values can roll forward.

The current codebase stores canonical investment facts in SQLite and already propagates some changes through the
in-process queue. However, `tax-reporting` is currently read-only and only generates annual assets output on demand.
There is no persisted monthly artifact, no month history cache, and no background recalculation for derived tax views.

The user approved two technical directions during TechSpec clarification:

- Persist a month-level calculation artifact instead of recomputing history entirely on demand.
- Recalculate affected months automatically when upstream facts change, while still showing what changed.

## Decision

V1 will persist one monthly close artifact per month and update it automatically when upstream facts change.

The implementation will:

- Add a `monthly_tax_closes` table owned by `tax-reporting`.
- Store one current artifact row per month, keyed by `YYYY-MM`, with summary columns for history rendering and a JSON
  audit payload for month detail.
- Persist at least: month, workspace state, outcome, calculation version, input fingerprint, calculated timestamp,
  net tax due, carry-forward out, change summary, and detail payload.
- Recalculate forward from the earliest affected year whenever canonical facts change, because carry-forward effects can
  alter later months.
- Subscribe monthly recalculation to existing queue events for imported transactions, consolidated position imports, and
  transaction fee reallocations.
- Add a new asset-tax-classification change event so monthly close also refreshes automatically after asset type repairs.
- Overwrite the current month artifact in place rather than persisting a full revision ledger in V1.

## Alternatives Considered

### Alternative 1: Recompute Monthly History Entirely on Demand

- **Description**: Build month history and month detail directly from transactions every time the workspace opens.
- **Pros**: No new persistence table, fewer migrations, simpler write path.
- **Cons**: Weak audit boundary, slower history rendering, no stable change summary, harder to explain why a month
  changed after upstream edits.
- **Why rejected**: It does not meet the approved trust and explainability requirements cleanly.

### Alternative 2: Persist a Full Calculation Ledger and Revision History

- **Description**: Save per-operation tax lines, intermediate monthly steps, and historical revisions for every month.
- **Pros**: Maximum audit depth, easier historical diffing, strong future compliance story.
- **Cons**: Larger schema, more write volume, more migration and maintenance cost, slower MVP path.
- **Why rejected**: It exceeds V1 needs. The approved scope calls for a minimal persisted month artifact, not a full
  accounting ledger.

### Alternative 3: Mark Months Obsolete and Require Manual Recalculation

- **Description**: Keep persisted artifacts, but only invalidate them when facts change and wait for explicit user
  recalculation.
- **Pros**: Lower background work, simpler queue integration.
- **Cons**: More user friction, weaker freshness, worse fit for the approved automatic recalculation direction.
- **Why rejected**: The user explicitly chose automatic recalculation.

## Consequences

### Positive

- History loading becomes fast and deterministic because the workspace reads persisted month summaries.
- Month detail gains a stable audit payload that can explain outcomes without re-deriving everything in the renderer.
- Automatic recalculation keeps monthly history current after upstream fixes.
- Change summaries can be surfaced directly in history after a recomputation.

### Negative

- `tax-reporting` now needs its own persistence and queue-handling concerns.
- Artifact invalidation and forward recomputation increase implementation complexity.
- The table stores derived data that must stay consistent with canonical upstream facts.

### Risks

- **Drift risk**: persisted artifacts could diverge from canonical inputs if recalculation misses a change path.
  Mitigation: route all supported upstream mutations through queue-driven recalculation or explicit recalculation use
  cases, and cover them with integration tests.
- **Schema lock-in risk**: the initial artifact shape may be too narrow for future tax coverage.
  Mitigation: keep summary columns minimal and place extensible audit detail inside versioned JSON payloads.
- **Recalculation fan-out risk**: one upstream change can affect many later months.
  Mitigation: recompute from the earliest affected year forward and keep history reads artifact-based.

## Implementation Notes

- The artifact table should optimize the history query first; month detail can read the JSON payload for deeper audit.
- Automatic recalculation should reuse the existing in-process queue pattern so upstream IPC commands can complete only
  after dependent monthly updates have finished.
- The first migration should introduce only the current-state artifact table. Revision history is explicitly deferred.
