# ADR-006: Keep Monthly Repair Read-Only and Route Users to Existing Flows

## Status

Accepted

## Date

2026-05-07

## Context

The PRD requires blocked months to explain missing facts clearly and guide the user to the exact information needed to
continue. At the same time, the repository already has working flows for:

- importing and editing daily broker taxes in `ImportPage`
- repairing asset metadata and asset type in `AssetsPage`
- maintaining brokers in `BrokersPage`

Embedding those edits directly inside the monthly workspace would make `tax-reporting` mutate data owned by
`portfolio` and `ingestion`, and would add substantial UI scope to the first monthly close release.

During clarification, the user chose the narrowest repair scope for MVP: the monthly workspace should not add new
inline repair forms, and blocked months should instead guide users to existing screens.

## Decision

The monthly workspace will be read-only for data repair in MVP and will return structured repair targets that point to
existing app flows.

The implementation will:

- Keep canonical edits in their current bounded contexts.
- Have month detail return `blockedReasons` with plain-language messages plus a structured `repairTarget` containing at
  least a destination tab and a repair hint code.
- Add lightweight renderer navigation support so `MonthlyTaxPage` can switch the app to the correct tab when the user
  follows a repair action.
- Defer inline repair forms, modal editors, and tax-fact mutations inside the monthly workspace.

## Alternatives Considered

### Alternative 1: Inline Monthly Repair Forms

- **Description**: Add forms for IRRF, daily broker taxes, and asset metadata directly inside month detail.
- **Pros**: More self-contained flow, fewer context switches for the user.
- **Cons**: Cross-context writes from `tax-reporting`, larger renderer scope, more validation and duplication of
  existing forms.
- **Why rejected**: The user explicitly chose not to add new repair forms in MVP.

### Alternative 2: Hybrid Repair Flow

- **Description**: Keep some edits inline and redirect only the more complex ones.
- **Pros**: Balances convenience and scope.
- **Cons**: Still duplicates ownership and creates an inconsistent repair story.
- **Why rejected**: It adds complexity without a clear MVP need.

### Alternative 3: Plain Text Errors Without Navigation Metadata

- **Description**: Show blocked reasons, but leave the user to find the correct repair screen manually.
- **Pros**: Lowest implementation cost.
- **Cons**: Poor guidance, weak trust recovery path, worse fit for the PRD.
- **Why rejected**: The PRD explicitly requires guided recovery for blocked months.

## Consequences

### Positive

- Bounded-context ownership stays intact.
- MVP UI scope stays focused on month history and month explanation.
- Existing import and asset maintenance flows gain more value instead of being bypassed.

### Negative

- Users must switch tabs to fix some problems.
- The renderer needs a small cross-page navigation mechanism that does not exist today.

### Risks

- **Context-switch risk**: users may lose track of what they are fixing.
  Mitigation: include month, ticker, and reason details in the repair CTA and refresh monthly history automatically when
  the user returns.
- **Navigation ambiguity risk**: some blocked reasons may not map to a single page cleanly.
  Mitigation: define a small set of repair target codes and keep the first version intentionally narrow.

## Implementation Notes

- Because the app uses tab state rather than route URLs, repair targets should be tab-oriented, not router-oriented.
- The first version can support tab switching plus a hint banner. Deep-link scrolling or auto-opening a specific modal
  is optional and can remain out of scope.
- The monthly detail contract should keep repair target metadata separate from human-readable copy so the renderer can
  evolve the UI without changing domain messages.
