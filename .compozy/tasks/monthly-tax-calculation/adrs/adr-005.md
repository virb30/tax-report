# ADR-005: Keep Monthly Close in Tax Reporting with Coarse-Grained IPC

## Status

Accepted

## Date

2026-05-07

## Context

The repository already separates canonical portfolio facts from derived tax reporting:

- `portfolio` owns positions, transactions, brokers, assets, and recalculation of holdings.
- `ingestion` owns import flows and daily broker tax persistence.
- `tax-reporting` currently owns annual report generation as a derived workflow.

Monthly close is also a derived tax workflow, but unlike the annual report it needs its own repository, queue
subscriptions, and IPC contracts. It depends on canonical data from multiple contexts, especially transactions, asset
classification, broker metadata, and daily broker taxes/IRRF.

The user approved two architectural directions during clarification:

- Monthly close should live inside `src/main/tax-reporting`.
- The renderer should use a coarse-grained IPC surface centered on history, month detail, and recalculation.

## Decision

Monthly close will be implemented inside `src/main/tax-reporting` as a new derived-tax slice with coarse-grained IPC
contracts.

The implementation will:

- Extend `tax-reporting` to own monthly calculation services, monthly artifact repositories, and queue handlers.
- Keep `portfolio` as the source of truth for transactions, positions, assets, and brokers.
- Read daily broker taxes from `ingestion` through an injected repository dependency.
- Extend `createTaxReportingModule(...)` so it receives `shared` infrastructure plus the required `portfolio` and
  `ingestion` exports.
- Expose a small renderer contract set:
  - `listMonthlyTaxHistory`
  - `getMonthlyTaxDetail`
  - `recalculateMonthlyTaxHistory`
- Keep detailed audit assembly in the main process. The renderer consumes ready-to-render summaries and month detail
  payloads rather than composing tax logic locally.

## Alternatives Considered

### Alternative 1: Implement Monthly Close Inside Portfolio

- **Description**: Put monthly close inside `portfolio` because it depends heavily on transactions and positions.
- **Pros**: Fewer module boundaries, easier direct access to portfolio internals.
- **Cons**: Mixes canonical holdings concerns with derived tax workflows and broadens `portfolio` responsibility.
- **Why rejected**: Monthly close is a tax-reporting concern and fits the existing derived-output direction better.

### Alternative 2: Extract a Shared Financial Calculation Core First

- **Description**: Create a new cross-context calculation module for both portfolio and tax reporting before building the
  monthly feature.
- **Pros**: Cleaner long-term separation, possible reuse across annual and monthly outputs.
- **Cons**: Larger refactor, more abstractions before the behavior is proven, higher delivery cost.
- **Why rejected**: It adds architectural ceremony before the MVP monthly workflow is validated.

### Alternative 3: Use Fine-Grained IPC Contracts Per Panel

- **Description**: Split history, blocked reasons, group summaries, disclosures, and carry-forward data into many IPC
  calls.
- **Pros**: Smaller payloads, more independent panel loading.
- **Cons**: More IPC surface, more renderer orchestration, higher coupling between page structure and backend API.
- **Why rejected**: The user chose a coarse-grained monthly workspace contract.

## Consequences

### Positive

- The feature stays aligned with the existing derived-tax boundary of `tax-reporting`.
- The renderer remains simple because tax logic and aggregation stay in the main process.
- IPC contracts stay small and stable enough to support the tabbed Electron UI pattern already used in the app.

### Negative

- `tax-reporting` module composition becomes more complex than the current annual-report-only implementation.
- The app container types need to be expanded to pass `shared` and `ingestion` dependencies into `tax-reporting`.

### Risks

- **Boundary leakage risk**: monthly close could start mutating canonical portfolio data directly.
  Mitigation: keep it read-only over portfolio and ingestion repositories; only its own artifact repository may write.
- **Contract bloat risk**: month detail payloads could become too UI-specific.
  Mitigation: keep the contract organized around stable domain concepts such as summary, groups, blocked reasons, and
  disclosures.

## Implementation Notes

- `tax-reporting` should introduce a monthly history use case, a month detail use case, and a recalculation use case.
- The module should register its own startup handler for queue subscriptions, similar to `portfolio`'s background
  recalculation handler.
- The IPC contracts should live under `src/ipc/contracts/tax-reporting/monthly-close/`.
