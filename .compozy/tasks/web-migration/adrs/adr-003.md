# ADR-003: Keep Server-Side SQLite for Phase 1

## Status

Accepted

## Date

2026-05-13

## Context

The current persistence layer uses Knex with `better-sqlite3`. Database migrations live in code, repositories already
depend on Knex, and the Electron runtime derives the database path from the desktop user data directory.

The web migration must replace desktop distribution first. PostgreSQL is a better long-term fit for hosted multi-user
operation, but moving to PostgreSQL at the same time as the runtime migration would increase Phase 1 risk and scope.

## Decision

Phase 1 will keep server-side SQLite behind the existing Knex repository interfaces. The web backend will own the
SQLite database path through server configuration instead of Electron `userData`. PostgreSQL migration is deferred to a
separate future initiative.

## Alternatives Considered

### Alternative 1: PostgreSQL in Phase 1

- **Description**: Replace SQLite with PostgreSQL during the web migration.
- **Pros**: Better hosted web fit and clearer path toward multi-user operation.
- **Cons**: Adds database dialect changes, deployment provisioning, migration validation, and repository risk to the
  initial web migration.
- **Why rejected**: It expands Phase 1 beyond the desktop-to-web runtime migration.

### Alternative 2: Browser-local storage

- **Description**: Store workflow state in IndexedDB or another browser-local store.
- **Pros**: Avoids server persistence and hosting database setup.
- **Cons**: Conflicts with the selected HTTP backend architecture and weakens server-side privacy and consistency
  controls.
- **Why rejected**: It would force persistence behavior into the browser instead of preserving backend use case
  boundaries.

## Consequences

### Positive

- Existing Knex repositories and migrations remain usable.
- Phase 1 avoids combining runtime migration with database engine migration.
- The migration can focus verification on workflow parity and transport behavior.

### Negative

- The first web release inherits SQLite deployment constraints.
- PostgreSQL-readiness remains future work.
- Server configuration must replace Electron-specific database path resolution.

### Risks

- Hosted SQLite has concurrency and operational limits that must be documented for Phase 1.
- Future PostgreSQL migration may expose SQLite-specific SQL assumptions in repositories or migrations.

## Implementation Notes

Introduce a server runtime configuration object for database path resolution. Do not keep `ElectronRuntime.getUserDataPath`
as the source of database ownership for the web backend.
