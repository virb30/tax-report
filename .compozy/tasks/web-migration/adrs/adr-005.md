# ADR-005: Parse Browser File Uploads in the Backend

## Status

Accepted

## Date

2026-05-13

## Context

The desktop import flow opens an Electron file dialog, passes a local file path through IPC, and lets backend ingestion
code read the file through SheetJS. Browser code cannot pass trusted local file paths to the backend. Web import must
use browser file selection and upload semantics while preserving preview and confirmation behavior.

The existing ingestion layer already owns CSV/XLSX parsing and import validation behavior, so moving parsing into the
browser would duplicate trust-sensitive logic.

## Decision

The browser will upload CSV/XLSX files to the Express backend using multipart requests. The backend will parse uploaded
files and return preview or import confirmation results. The import UI will keep the user-facing preview and confirmation
steps required by the PRD.

## Alternatives Considered

### Alternative 1: Frontend parses files into rows

- **Description**: Use browser File APIs and SheetJS in the frontend, then send normalized rows to the backend.
- **Pros**: Avoids multipart upload handling and can provide immediate client-side feedback.
- **Cons**: Duplicates parsing trust at the client boundary and moves ingestion responsibility into the renderer.
- **Why rejected**: It weakens the backend ingestion boundary and creates two parsing surfaces.

### Alternative 2: Temporary upload sessions

- **Description**: Upload once, store a temporary import session on the backend, and preview or confirm by session ID.
- **Pros**: Avoids repeated file upload between preview and confirmation and handles larger files better.
- **Cons**: Adds temporary storage, cleanup, expiration, and state management.
- **Why rejected**: It is unnecessary for Phase 1 unless file size or performance testing proves repeated upload is a
  problem.

## Consequences

### Positive

- Existing backend ingestion parsing remains the authority.
- Browser import follows normal file upload expectations.
- The UI can preserve preview and confirmation without exposing local paths.

### Negative

- The backend needs multipart handling, upload limits, and file type validation.
- Preview and confirmation may upload the file twice unless a later task introduces temporary sessions.
- Existing file reader interfaces need to support uploaded buffers or temporary server files.

### Risks

- Large files may make repeated upload inefficient.
- Temporary server files could leak sensitive data if cleanup fails.
- File validation must not rely only on browser-provided filenames or MIME types.

## Implementation Notes

Keep upload handling in HTTP transport code. Convert uploaded file data into an ingestion input accepted by application
use cases. If implementation chooses temporary files, ensure cleanup occurs after each request and on error paths.
