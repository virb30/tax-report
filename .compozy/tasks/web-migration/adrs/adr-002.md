# ADR-002: Use React with an Express HTTP API

## Status

Accepted

## Date

2026-05-13

## Context

Tax Report currently runs as an Electron desktop application. The renderer calls `window.electronApi`, preload exposes
IPC methods, and the main process initializes SQLite, module containers, repositories, use cases, and IPC handlers.

The web migration must preserve the existing core workflows while removing the desktop runtime from the primary
distribution path. The current business logic is already organized into application use cases and infrastructure
repositories, so the migration can reuse the backend behavior if a new transport adapter replaces Electron IPC.

## Decision

Phase 1 will use the existing React renderer as the browser client and add an Express-based Node HTTP API as the web
backend. The HTTP API will call the existing application use cases through web transport handlers. The renderer will
move from `window.electronApi` calls to browser API services that use HTTP.

Electron IPC remains a desktop transport concern only while migration work is active. It will not be the foundation for
the web runtime.

## Alternatives Considered

### Alternative 1: Browser-only local app

- **Description**: Move persistence, parsing, and workflow state into the browser.
- **Pros**: Avoids backend deployment and keeps the product install-free.
- **Cons**: Requires a larger rewrite, weakens centralized data handling, and conflicts with hosted web expectations.
- **Why rejected**: It does not reuse the existing backend use cases effectively and makes fiscal data reliability
  harder to control.

### Alternative 2: Electron backend bridge for web

- **Description**: Preserve the Electron-style API bridge and adapt it for browser access.
- **Pros**: Minimizes near-term renderer changes.
- **Cons**: Carries Electron-specific semantics into the web product and keeps runtime coupling that the migration is
  meant to remove.
- **Why rejected**: It preserves desktop architecture constraints instead of establishing a web architecture.

### Alternative 3: Fastify HTTP API

- **Description**: Use Fastify instead of Express for the HTTP backend.
- **Pros**: Strong schema integration and good performance.
- **Cons**: Adds framework-specific conventions that are not needed for the first migration.
- **Why rejected**: Express is sufficient for the current API surface and keeps the adapter simple.

## Consequences

### Positive

- Existing application use cases and repositories remain reusable.
- The migration creates a clear transport boundary between browser UI and backend behavior.
- Express supports the required API and multipart file upload needs with limited framework complexity.

### Negative

- The renderer needs an API service abstraction instead of direct `window.electronApi` usage.
- The backend needs a new composition root and route registration layer.
- The project will temporarily contain both IPC and HTTP transport code during the migration.

### Risks

- Route handlers may duplicate logic from IPC handlers if transport boundaries are not kept thin.
- Existing inconsistent IPC result/error semantics may leak into HTTP unless the web API defines consistent status and
  error mapping.

## Implementation Notes

Add web transport handlers outside the renderer and application layers. Keep handlers responsible for HTTP parsing,
validation, status mapping, and use case invocation only.
