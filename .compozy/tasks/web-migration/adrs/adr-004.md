# ADR-004: Keep Backend and Frontend Schemas Independent

## Status

Accepted

## Date

2026-05-13

## Context

The Electron implementation shares IPC contracts and output types between main process, preload, and renderer code.
That works inside one desktop repository, but it couples frontend and backend evolution through shared TypeScript
contracts.

The web product should be designed as if the frontend and backend can evolve independently, including the possibility
that they later live in separate repositories.

## Decision

The web migration will not create a shared frontend/backend schema package for HTTP contracts. The backend will own
request validation and response mapping at the HTTP boundary. The frontend will own its API client response types and
runtime parsing where needed.

The backend may reuse existing IPC Zod schemas internally as implementation references during migration, but those
schemas will not become shared browser contracts.

## Alternatives Considered

### Alternative 1: Shared REST contract package

- **Description**: Move HTTP DTO schemas to a shared package imported by frontend and backend.
- **Pros**: Reduces duplicate type definitions and keeps compile-time alignment.
- **Cons**: Couples release cadence and assumes a monorepo/shared-package deployment model.
- **Why rejected**: The desired architecture treats frontend and backend schemas as independently evolvable.

### Alternative 2: Reuse IPC contracts as-is

- **Description**: Use current `src/ipc` contracts directly for HTTP.
- **Pros**: Fastest initial migration path.
- **Cons**: Leaks Electron channel metadata and IPC error modes into the web API.
- **Why rejected**: It preserves desktop transport semantics instead of defining an HTTP boundary.

## Consequences

### Positive

- Frontend and backend can evolve with explicit API compatibility checks instead of shared type coupling.
- The HTTP API can define web-specific status codes, error bodies, and upload semantics.
- Future repository separation remains possible.

### Negative

- Some DTO shape duplication is intentional.
- API compatibility must be enforced through tests and documented contracts instead of shared compile-time types.
- The renderer API client needs its own type and parsing layer.

### Risks

- Backend and frontend schemas may drift without API integration and browser workflow tests.
- Migration work may be slower because schemas must be mapped deliberately rather than imported.

## Implementation Notes

Add backend route validators near route handlers. Add frontend API client types in renderer services. Use integration
tests to verify request/response compatibility for every parity workflow.
