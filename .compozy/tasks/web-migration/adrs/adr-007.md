# ADR-007: Split Backend and Frontend into Independent Projects

## Status

Accepted

## Date

2026-05-13

## Context

The current Electron repository has one `package.json`, one dependency tree, one source root, and shared build/test
configuration for main, preload, renderer, and IPC code. The web migration changes the product shape to a browser
frontend and an HTTP backend.

The new structure should make backend and frontend operate like independent projects. The repository root should hold
LLM rules, skills, README, and general orientation only. Backend-specific and frontend-specific dependencies,
configuration, tests, build scripts, and source code should live inside their respective project directories.

## Decision

Tax Report will not treat the web migration repository as a monorepo workspace. It will contain two independent project
directories:

- `backend/` with its own `package.json`, `node_modules`, configuration, tests, and `src/`.
- `frontend/` with its own `package.json`, `node_modules`, configuration, tests, and `src/`.

All project-specific source code must live under the corresponding `src` directory, such as `backend/src/...` and
`frontend/src/...`. The repository root must avoid backend/frontend-specific build, lint, test, and dependency
configuration after the migration structure is introduced.

## Alternatives Considered

### Alternative 1: Monorepo workspace

- **Description**: Use a root workspace with shared package management and shared scripts for backend and frontend.
- **Pros**: Centralized commands and easier dependency deduplication.
- **Cons**: Treats backend and frontend as coupled packages and conflicts with the desired independent-project model.
- **Why rejected**: The repository must not be considered a monorepo for this migration.

### Alternative 2: Keep current root-level Electron project

- **Description**: Add web backend and frontend code under the existing root project.
- **Pros**: Smallest immediate structural change.
- **Cons**: Preserves desktop-era coupling, shared dependency scope, and root-level project configuration.
- **Why rejected**: It does not make backend and frontend independent projects.

## Consequences

### Positive

- Backend and frontend dependencies, scripts, and configuration can evolve independently.
- The root directory becomes simpler and focused on repository-level guidance.
- Future physical repository separation remains straightforward.

### Negative

- Shared compile-time TypeScript imports between backend and frontend are intentionally unavailable.
- Developers must run project-specific commands from `backend/` or `frontend/`.
- Migration requires moving configuration and updating paths in tests, imports, and build scripts.

### Risks

- Accidental root-level project configuration may reintroduce monorepo coupling.
- Moving code can break relative imports and test configuration if not sequenced carefully.

## Implementation Notes

Move backend code into `backend/src`, frontend code into `frontend/src`, and keep project-specific `package.json`,
`tsconfig`, lint, test, format, and build configuration inside each project directory. Keep root files limited to README,
LLM rules, workflow documentation, and repository-level orientation.
