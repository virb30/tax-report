# ADR-006: Require Unit, API Integration, and Browser E2E Verification

## Status

Accepted

## Date

2026-05-13

## Context

The PRD requires desktop replacement only after the web product supports import, initial balance, positions, monthly
tax, annual report, assets, and brokers with equivalent outcomes. The current project uses Jest for main and renderer
tests and has renderer-level E2E-style tests that mock Electron APIs.

Replacing Electron IPC with HTTP changes transport behavior, browser file import, and user workflow execution. Unit
tests alone cannot prove that the web product replaces the desktop workflow.

## Decision

Phase 1 verification will include unit tests for domain and application behavior, HTTP API integration tests for route
validation and use case integration, and browser E2E coverage for the seven parity workflows before desktop distribution
is replaced.

## Alternatives Considered

### Alternative 1: Unit and API integration tests only

- **Description**: Validate backend behavior and HTTP routes without browser workflow automation.
- **Pros**: Faster to implement and less infrastructure-heavy.
- **Cons**: Does not prove users can complete the PRD workflows in the browser.
- **Why rejected**: It leaves the central replacement criterion under-tested.

### Alternative 2: Browser E2E first

- **Description**: Prioritize browser flows and rely on them for most confidence.
- **Pros**: Tests user-visible behavior directly.
- **Cons**: Makes fiscal calculation regressions harder to diagnose and can miss backend edge cases.
- **Why rejected**: It gives weak feedback for domain and route-level failures.

## Consequences

### Positive

- Verification aligns with the PRD parity requirement.
- API contract drift is caught through integration tests despite independent frontend and backend schemas.
- Browser workflows become the release gate for desktop replacement.

### Negative

- Test setup expands beyond the current Jest-only workflow.
- Browser E2E data setup must be deterministic.
- Full verification will take longer than unit tests alone.

### Risks

- E2E tests can become brittle if selectors and test data are not designed deliberately.
- API and browser tests may duplicate setup unless shared fixtures are planned.

## Implementation Notes

Keep existing Jest unit coverage. Add API integration tests against the Express app without starting a network listener
where possible. Add browser E2E tests for import, initial balance, positions, monthly tax, annual report, assets, and
brokers as the desktop replacement gate.
