# ADR-001: Replace Desktop Distribution with a Web Product

## Status

Accepted

## Date

2026-05-13

## Context

Tax Report is currently implemented as an Electron desktop application, but it has no current user base that depends on
the local desktop product. The desktop implementation creates product and operational cost around platform-specific
packaging, native dependency rebuilds, Electron runtime behavior, IPC-specific test setup, and user installation. The
product direction is to simplify maintenance, tests, and distribution while preserving the intended core user workflows.

The selected migration should keep continuity for the planned product capabilities, but it does not need to account for
existing users or existing local desktop data. The web experience may change where browser patterns make the product
simpler, especially file import and navigation.

## Decision

Tax Report will move to a web product that replaces the desktop application for new distribution. The first web release
must provide functional parity for the core workflows: import, initial balance, positions, monthly tax, annual report,
assets, and brokers.

The first web release will not include a desktop data migration path because there are no existing users to migrate.
After the web version covers the core workflows, distribution will focus on the web product and desktop distribution
will end.

## Alternatives Considered

### Alternative 1: Minimal Web Release Focused on Internal Efficiency

- **Description**: Release a smaller web version covering only the most critical fiscal workflows and defer some
  administrative or support screens.
- **Pros**: Faster initial delivery and earlier reduction of some desktop maintenance costs.
- **Cons**: Users would not receive an equivalent product, and the team would need to maintain two incomplete surfaces
  during the transition.
- **Why rejected**: It conflicts with the requirement to preserve functional continuity across all central workflows.

### Alternative 2: Long Parallel Desktop and Web Transition

- **Description**: Keep desktop and web products active in parallel while web parity is built gradually.
- **Pros**: Lower disruption for existing desktop users and more time to validate web behavior.
- **Cons**: Maintains duplicated distribution, test, and support burden for longer.
- **Why rejected**: It preserves the operational complexity that the migration is meant to remove.

## Consequences

### Positive

- Distribution becomes centered on browser access instead of platform-specific installers.
- The product has one forward-looking surface for new users.
- Tests can focus on web product behavior rather than Electron-specific runtime contracts.
- The team can simplify support expectations around installation, updates, and local runtime differences.

### Negative

- The first release requires broad functional coverage before desktop distribution can end.
- The team will not spend Phase 1 effort on migration paths from a product surface that has no current users.

### Risks

- Parity across all core workflows may increase the first release size.
- Browser-based import and navigation changes may require validation during onboarding.

## Implementation Notes

Detailed implementation choices belong in the TechSpec. The PRD should define user-facing parity, transition policy,
and success criteria without prescribing the web architecture.
