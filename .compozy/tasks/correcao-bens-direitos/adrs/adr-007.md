# ADR-007: Keep Unsupported Import Issues Ephemeral in the MVP

## Status

Accepted

## Date

2026-04-28

## Context

The MVP must surface unsupported asset classes and events clearly during import, but the user chose not to persist unsupported-line history beyond the import session. The current application has no import-issues store, and adding one would introduce a new persistence concept whose value is limited for the first release.

The product still needs to remain honest about scope in the report and catalog flows, but that can be satisfied through explicit scope messaging and import-time review rather than a durable skipped-lines ledger.

## Decision

Unsupported transaction and consolidated-position lines will remain ephemeral in the MVP.

The system will:

- surface unsupported lines and skipped counts during preview and confirm;
- block those lines from entering supported-data persistence;
- persist only accepted supported data and any catalog overrides tied to it;
- show general supported-scope messaging in report and maintenance flows without a historical unsupported-issues store.

## Alternatives Considered

### Alternative 1: Persist a Lightweight Import-Issues Store

- **Description**: Save skipped unsupported lines so later flows can surface them again.
- **Pros**: Better historical visibility and easier follow-up after import.
- **Cons**: Adds schema, repositories, UI states, and lifecycle rules for limited MVP payoff.
- **Why rejected**: It broadens scope into issue-tracking infrastructure that the MVP does not need.

### Alternative 2: Block Confirmation When Unsupported Lines Exist

- **Description**: Refuse confirmation until the file contains only supported lines.
- **Pros**: Strong guardrail and simpler downstream assumptions.
- **Cons**: Prevents users from getting value from supported holdings in mixed files.
- **Why rejected**: It conflicts with the approved partial-processing product approach.

## Consequences

### Positive

- No new persistence model is required for skipped import lines.
- Import review stays focused on the current file and current correction decisions.
- Supported data can proceed without carrying a durable backlog concept.

### Negative

- Users must act on unsupported-line feedback during the import session or revisit the source file later.
- Later report views cannot enumerate previously skipped unsupported rows.

### Risks

- Users may forget unsupported items after leaving the import flow.
- Mitigation: make preview/confirm feedback explicit, include skipped counts in import results, and keep scope messaging visible in report and catalog screens.

## Implementation Notes

- Extend preview and confirm result contracts with unsupported-line summaries.
- Do not add a new database table for import issues in this MVP.
- Keep unsupported scope warnings consistent between transaction import and consolidated-position import.
