# ADR-002: Continue with Supported Data and Surface Unsupported Lines as Pending Work

## Status

Accepted

## Date

2026-04-28

## Context

The current supported scope for Bens e Direitos is intentionally narrower than the full range of Brazilian investment assets and corporate events. Users still need visibility into unsupported lines, but a hard-stop experience would reduce the product's usefulness for investors whose portfolio is mostly supported.

The chosen product direction is to produce trustworthy declaration-ready output for supported cases without pretending that unsupported lines are complete.

## Decision

When the product encounters unsupported asset classes or unsupported events, it will continue processing supported data and will place unsupported lines in a visible pending state. The annual report may still be generated in partial mode, provided the product clearly distinguishes:

- items that are ready to copy;
- items that are pending user action or missing metadata;
- lines that are unsupported and therefore excluded from declaration-ready output.

## Alternatives Considered

### Alternative 1: Block the Entire Workflow

- **Description**: Stop import or report generation as soon as any unsupported line is detected.
- **Pros**: Simple mental model; no risk that users overlook unsupported lines.
- **Cons**: Punishes users whose supported data could already be handled correctly; creates unnecessary friction for mixed portfolios.
- **Why rejected**: It is too rigid for the selected self-service user and reduces practical value.

### Alternative 2: Best-Effort Output with Warnings Only

- **Description**: Process everything and rely on warnings without strong pending-state separation.
- **Pros**: Lowest friction; fastest perceived flow.
- **Cons**: Too easy to confuse incomplete output with declaration-ready output.
- **Why rejected**: It weakens trust and conflicts with the product's declaration-ready positioning.

## Consequences

### Positive

- Users still receive useful output for supported holdings.
- Unsupported scope becomes visible and actionable instead of silent.
- The product can stay honest about what it does and does not support.

### Negative

- The UI must explain multiple item states clearly.
- Some users may expect full completion once they see a report, even if it is partial.

### Risks

- Users may miss pending or unsupported items if status language is weak.
- Mitigation: use explicit status labels, disable copy for non-ready items, and summarize pending counts prominently.

## Implementation Notes

This ADR defines product behavior and communication only. Detailed status taxonomy and user flows belong in the TechSpec stage.
