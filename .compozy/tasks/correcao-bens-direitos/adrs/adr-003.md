# ADR-003: Extend `ticker_data` into the Canonical Asset Catalog

## Status

Accepted

## Date

2026-04-28

## Context

The current `ticker_data` table stores only `ticker`, required `cnpj`, and optional `name`. The Bens e Direitos correction scope needs one canonical place to persist supported asset type, issuer metadata completeness, and the minimal provenance needed to avoid repeated review prompts during import and reporting.

The repository already exposes `KnexAssetRepository` and the application already depends on it in report generation. Adding a second table for type resolution would increase joins and coordination without solving a distinct domain need in this MVP.

## Decision

`ticker_data` will become the canonical asset catalog for the MVP.

The table will be extended to:

- allow `cnpj` to be nullable;
- keep `name` nullable;
- add nullable `asset_type`;
- add a persisted `resolution_source` field for asset type provenance;
- keep the table keyed by `ticker`.

The application will use this catalog as the single persisted source for:

- supported asset type remembered across imports;
- issuer metadata used by the annual report;
- catalog completeness queries used by review and pending flows.

## Alternatives Considered

### Alternative 1: Separate Asset Type Resolution Table

- **Description**: Keep `ticker_data` for issuer metadata only and add a second table for type and override state.
- **Pros**: More explicit separation of concerns.
- **Cons**: More joins, more repositories, more migration and IPC surface, and no clear product benefit for the MVP.
- **Why rejected**: The split adds complexity without representing a genuinely separate business object.

### Alternative 2: Persist Only Asset Type and Derive Everything Else Transiently

- **Description**: Store only canonical asset type and compute review provenance entirely at runtime.
- **Pros**: Smaller schema change.
- **Cons**: Harder to explain why a type was trusted, and harder to avoid repeated review prompts.
- **Why rejected**: The MVP needs minimal persisted provenance to support review and repair flows coherently.

## Consequences

### Positive

- One catalog supports import, repair, reporting, and maintenance flows.
- Manual corrections persist once and become reusable across later operations.
- The renderer and IPC layers can treat catalog state as the authoritative reference.

### Negative

- `ticker_data` becomes broader than a simple issuer-metadata table.
- Existing rows and tests must adapt to nullable issuer fields and new catalog fields.

### Risks

- Catalog semantics may drift if future features overload the table with unrelated concerns.
- Mitigation: keep the catalog limited to ticker identity, supported asset type, issuer metadata, and type-resolution provenance.

## Implementation Notes

- Add a new database migration after `012-add-active-to-brokers`.
- Extend the domain `Asset` entity and `AssetRepository` port before adding UI or IPC changes.
- Represent `resolution_source` with the same supported states used by the MVP review flows.
