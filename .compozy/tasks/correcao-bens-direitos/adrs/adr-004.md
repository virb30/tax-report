# ADR-004: Use Preview-Confirm Contracts as the Import Review Boundary

## Status

Accepted

## Date

2026-04-28

## Context

The current import flows split work into preview and confirm, but the contracts do not carry asset-type review data. Preview returns parsed rows only, and confirm accepts only `filePath` or `filePath + year`. That structure cannot support in-flow manual overrides for unresolved types, nor can it surface unsupported lines as first-class pending work while preserving a coherent confirmation experience.

The renderer already has page-local hooks for transaction import and consolidated-position import. Extending the existing preview-confirm contract pattern is lower risk than forcing users into a separate maintenance workflow before every import.

## Decision

The technical review boundary for imports will remain the preview-confirm contract pair.

Preview responses will be extended to include:

- resolved asset type per supported row when known;
- resolution status and review requirements per ticker or row;
- unsupported-line information and pending summaries needed by the renderer.

Confirm commands will be extended to accept explicit `assetTypeOverrides` keyed by ticker. The main process will persist accepted overrides into the asset catalog before saving transactions or initial-balance transactions derived from consolidated imports.

## Alternatives Considered

### Alternative 1: Force Users to Leave Import and Edit the Catalog First

- **Description**: Preview shows unresolved rows, but confirmation is blocked until the user updates the asset catalog elsewhere.
- **Pros**: Smaller import contract change.
- **Cons**: Breaks workflow continuity and makes the most common correction path slower.
- **Why rejected**: It pushes routine review work out of the import flow where the ambiguity is first visible.

### Alternative 2: Hard-Fail Import Without In-Flow Overrides

- **Description**: Any unresolved type aborts confirmation with no override payload.
- **Pros**: Simplest backend behavior.
- **Cons**: High friction and poor fit for the chosen declaration-preparation workflow.
- **Why rejected**: The MVP needs guided correction, not binary failure, for supported holdings.

## Consequences

### Positive

- Import remains the natural place to resolve type ambiguity.
- The backend receives explicit user decisions instead of guessing.
- Supported rows can proceed while unsupported rows remain visible and non-ready.

### Negative

- Shared contracts, preload typing, renderer hooks, and IPC tests must all change together.
- Import preview payloads become richer and require more renderer state handling.

### Risks

- Contract drift between transaction import and consolidated import could create inconsistent review behavior.
- Mitigation: define shared resolution-status semantics and reuse them across both import surfaces.

## Implementation Notes

- Reuse the existing typed IPC contract system rather than introducing ad hoc channels.
- Keep parsing and resolution separate so tests can target each phase directly.
- Persist accepted overrides before downstream recalculation is triggered.
