# ADR-005: Model Editable Initial Balance as Replace-All `initial_balance` Transactions per `ticker + year`

## Status

Accepted

## Date

2026-04-28

## Context

Initial balance is currently written as a single-broker command that appends one `initial_balance` transaction and mutates the position directly. The MVP needs multi-broker edit, replace, and delete behavior while preserving the existing recalculation model, where transactions remain the source of truth for year-end position recovery.

The codebase already recalculates positions from transactions, and consolidated-position import already deletes old `initial_balance` transactions per `ticker + year` before recreating them. That existing pattern can be generalized for the manual initial-balance workflow.

## Decision

The system will keep `transactions` as the source of truth for initial balance.

Editable initial balance will be implemented as a document-oriented application flow that:

- accepts one payload per `ticker + year` with shared fields and `allocations[]`;
- deletes existing `initial_balance` transactions for that `ticker + year`;
- recreates one `initial_balance` transaction per allocation;
- recalculates and saves the resulting position;
- supports read and delete flows derived from existing `initial_balance` transactions.

No separate `initial_balance_documents` table will be added in the MVP.

## Alternatives Considered

### Alternative 1: Dedicated Initial Balance Document Table

- **Description**: Persist document state separately and derive transactions from it.
- **Pros**: Clear document identity and easier audit metadata.
- **Cons**: More schema and synchronization work; duplicate sources of truth in the MVP.
- **Why rejected**: The existing transaction model already captures the data needed for recalculation and edit semantics.

### Alternative 2: Store Only Aggregate Initial Balance on Positions

- **Description**: Stop writing `initial_balance` transactions and mutate positions directly.
- **Pros**: Smaller write surface.
- **Cons**: Breaks recalculation consistency and makes historical reconstruction weaker.
- **Why rejected**: It would reintroduce the same fragility the MVP is trying to remove.

## Consequences

### Positive

- Reuses the current transaction-driven recalculation architecture.
- Makes save operations idempotent for `ticker + year`.
- Avoids a second persistence model for the same business concept.

### Negative

- Read flows must reconstruct document views from transaction groups.
- Deleting and rewriting allocations requires careful repository and test coverage.

### Risks

- Partial replacement bugs could corrupt broker allocations if delete and save are not treated as one logical flow.
- Mitigation: centralize replacement in one use case and cover it with repository integration tests.

## Implementation Notes

- Replace the current `setInitialBalance` command with a document payload rather than adding another parallel manual-base command.
- Add explicit list/get/delete use cases for initial-balance documents.
- Reuse year-scoped deletion helpers in `TransactionRepository`.
