# ADR-006: Assemble Annual Report Status On Demand and Reprocess Legacy Data Explicitly

## Status

Accepted

## Date

2026-04-28

## Context

The current annual report is built directly from saved positions and catalog metadata, with broker-level output and no concept of prior-year value, eligibility status, pending issues, or unsupported scope. The MVP also needs a safe way to repair legacy ticker types and then regenerate affected years without persisting stale report snapshots.

Persisting report status would create a second derived-state layer that must stay synchronized across imports, catalog edits, initial-balance edits, and legacy repairs. Renderer-side derivation would duplicate business rules across processes and weaken trust.

## Decision

The annual report will be assembled on demand in the main process by a dedicated report-assembly layer.

That layer will:

- compose year-end positions, prior-year reconstruction, broker data, and asset-catalog data;
- compute revenue classification, eligibility, pending issues, unsupported scope, and copy readiness at request time;
- emit one declaration item per supported `ticker + assetType`.

Legacy asset-type repair will be explicit and catalog-driven. Updating a ticker’s canonical type through the repair flow will trigger reprocessing of affected years on demand rather than refreshing persisted report snapshots.

## Alternatives Considered

### Alternative 1: Persist Report Status Snapshots

- **Description**: Save ready/pending/eligibility state in the database and refresh it after every change.
- **Pros**: Faster report reads and easier reporting on status counts.
- **Cons**: High synchronization cost and risk of stale derived data.
- **Why rejected**: The MVP already has many mutation paths; persisting snapshots would add failure modes without core user value.

### Alternative 2: Derive Status in the Renderer

- **Description**: Return raw report ingredients and let the UI compute statuses and pending logic.
- **Pros**: Smaller backend surface.
- **Cons**: Duplicates business rules outside the source-of-truth process and weakens IPC contract clarity.
- **Why rejected**: Declaration-readiness logic belongs in the main process with the domain data it depends on.

## Consequences

### Positive

- One backend path owns declaration-item assembly and status semantics.
- Legacy repair can reuse recalculation logic instead of inventing snapshot invalidation rules.
- IPC contracts can expose final report states directly to the renderer.

### Negative

- Report generation becomes more computation-heavy than the current flat position read.
- More application-layer services and tests are needed to compose prior-year and pending logic cleanly.

### Risks

- On-demand report generation may become slow if it recalculates too much history repeatedly.
- Mitigation: scope reprocessing to affected ticker-years and keep report assembly focused on the selected base year.

## Implementation Notes

- Build the new report assembly in the main process above repositories and below IPC handlers.
- Keep repair as an explicit use case rather than a hidden side effect of arbitrary catalog edits.
- Reuse existing recalculation services wherever they already represent the authoritative position logic.
