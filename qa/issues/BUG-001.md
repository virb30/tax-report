# BUG-001: Frontend dev server on `localhost:5173` does not route `/api` requests to the backend

**Impact (user-side):** Blocks-Completion
**Severity:** Critical
**Priority:** P0
**Type:** Functional
**Status:** pending
**Persona Affected:** New User
**Journey Step:** J-01 Import transactions and confirm preview, Step 2 (request preview)

## Environment

- **Build:** `8979c62`, local dev runtime; production-parity check failed
- **OS:** Linux 6.6.87.2-microsoft-standard-WSL2
- **Browser:** any browser reaching `http://localhost:5173`
- **Viewport:** n/a
- **Network:** wifi-fast
- **Locale:** pt-BR
- **URL:** `http://localhost:5173/api/health`

## Summary

From the user's perspective, the frontend shell is reachable at `localhost:5173`, but the same origin does not expose the backend API. A request to `/api/health` returns the Vite HTML shell instead of backend JSON, which means workflows that rely on relative `/api` calls cannot complete in this runtime.

## Reproduction

Charter: off-script
Tour: n/a

Steps:

1. Open the frontend dev server at `http://localhost:5173`.
2. Request a same-origin API path such as `http://localhost:5173/api/health`.
3. Observe the response body.

Observed:

- The response is `200 OK` with `Content-Type: text/html`.
- The body is the Vite HTML shell instead of backend JSON.
- Screenshot path: none; reproduced via HTTP response evidence.

## Expected

Same-origin `/api/*` requests from the frontend runtime should reach the backend API and return the corresponding JSON response, allowing browser workflows to complete their goals.

## Root cause

The frontend Vite dev configuration does not define a proxy or equivalent serving-layer integration for `/api`, while the frontend HTTP client defaults to relative `/api` requests.

## Fix

Pending. Either add a frontend dev proxy for `/api` to the backend origin or run the frontend behind a serving layer that already wires the backend under the same origin.
